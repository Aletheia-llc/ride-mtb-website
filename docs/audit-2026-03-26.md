# Ride MTB Codebase Audit Report

**Date:** 2026-03-26
**Auditor:** Claude Opus 4.6 (automated code review)
**Scope:** Full codebase at `/Users/kylewarner/Documents/ride-mtb`
**Stack:** Next.js 15.5.12, Prisma v7, PostgreSQL (Supabase), NextAuth v5, Tailwind CSS v4, Vercel

---

## Executive Summary

The Ride MTB codebase is a large-scale monolith spanning 23 feature modules with ~58 API routes and ~90 server actions. The overall architecture is well-organized with a clear `modules/` structure and good use of Prisma for most data access. However, the audit identified several significant concerns:

1. **A high-impact performance problem** in the NextAuth session callback that issues a database query on every single authenticated request.
2. **Inconsistent error boundary coverage** -- 21 route groups lack `error.tsx` and `loading.tsx`, meaning unhandled errors in those sections will crash the entire application shell.
3. **A classic N+1 query pattern** in the event import deduplication logic that will degrade severely at scale.
4. **27 duplicate "copy 2" files** cluttering the trails and forum modules, likely from accidental IDE duplication.
5. **Race conditions** in the marketplace watch/save toggle and credit tipping logic that can lead to incorrect counts under concurrent requests.

The codebase is in reasonable shape for an actively developed project, with good practices like Zod validation on server actions, parameterized SQL queries (no SQL injection found), and DOMPurify sanitization for rendered HTML. The issues below are ordered by severity to guide remediation priority.

## Critical Issues

### C1. Session callback queries the database on every authenticated request

**File:** `src/lib/auth/config.ts`, lines 41-53
**What:** The `session()` callback in the NextAuth config runs `db.user.findUnique()` on every request that touches the session. Since the root layout (`src/app/layout.tsx`) calls `await auth()` and has `export const dynamic = 'force-dynamic'`, this means every single page load triggers a database query to fetch the user's role, banned status, onboarding step, and email verification status.
**Why it matters:** This is the single highest-impact performance issue in the codebase. With JWT sessions, the session callback fires on every `auth()` call -- including on the root layout which wraps every page. Under load, this creates unnecessary database pressure proportional to total page views, not just auth-requiring pages.
**Suggested fix:** Cache the user role/status in the JWT token itself during the `jwt()` callback, and only refresh it periodically (e.g., every 5 minutes) or on specific events (role change, ban). The `jwt()` callback already has access to `user.id` -- extend it to also store role, banned status, etc. in the token payload.

### C2. Root layout `force-dynamic` disables all static optimization

**File:** `src/app/layout.tsx`, line 1
**What:** `export const dynamic = 'force-dynamic'` on the root layout forces every page in the entire application to be dynamically rendered on every request. No page can be statically generated or cached at the edge.
**Why it matters:** This is a global performance penalty. Pages like `/terms`, `/privacy`, `/register`, `/signin`, and marketing pages that have no user-specific content are being dynamically rendered unnecessarily. Combined with C1 (DB query per session call), every page load is both uncached and database-dependent.
**Suggested fix:** Remove `force-dynamic` from the root layout. Instead, move the `auth()` call into a separate component or use it only in pages/layouts that need it. Pages that need dynamic behavior can set `force-dynamic` individually.

### C3. Race condition in marketplace watch/save toggle

**File:** `src/app/api/marketplace/listings/[slug]/watch/route.ts`, lines 33-64
**What:** The watch toggle performs a read-then-write sequence (check if save exists, then create or delete + update count) without any transaction or row locking. Two concurrent requests can both read "no existing save" and both insert, or both read "existing" and both decrement the counter.
**Why it matters:** Under concurrent use, `saveCount` on listings can drift from the actual count of `ListingSave` records, producing incorrect UI data. This is a common bug pattern in toggle endpoints.
**Suggested fix:** Wrap the read-check-write sequence in a `db.$transaction()` or use an `upsert` pattern. Alternatively, compute `saveCount` from `_count` instead of maintaining a denormalized counter.

### C4. Event import dedup has N+1 query pattern

**File:** `src/modules/events/lib/import/dedup.ts`, lines 22-41
**What:** The `dedupAndInsert()` function loops over every imported event and performs two sequential database queries per event (one SELECT to check for duplicates, one INSERT if new). For a batch of 100 events, this means 100-200 individual database round trips.
**Why it matters:** This is a textbook N+1 problem. As the number of imported events grows, this function becomes progressively slower and puts unnecessary load on the database connection pool (which is only configured with `max: 2` connections).
**Suggested fix:** Batch the dedup: collect all `(importSource, externalId)` pairs, do a single `SELECT ... WHERE (importSource, externalId) IN (...)` to find existing ones, then batch-insert the new events. Alternatively, use a single `INSERT ... ON CONFLICT DO NOTHING` query.

## High Priority Issues

### H1. Env validation module (`src/lib/env.ts`) is defined but never imported

**File:** `src/lib/env.ts`
**What:** A thorough Zod-based environment validation schema exists but is never imported anywhere in the codebase. Searching for `from.*lib/env` yields zero results.
**Why it matters:** The validation is completely inert. The app boots successfully even when required env vars are missing, which can cause confusing runtime errors (e.g., `undefined` API keys, failed Stripe calls) rather than a clear startup failure. The `DATABASE_URL` and `DATABASE_DIRECT_URL` variable names in `env.ts` also don't match what `src/lib/db/client.ts` actually uses (`DATABASE_DIRECT_URL` and `DATABASE_POOLED_URL`).
**Suggested fix:** Import `env` in `src/lib/db/client.ts` and other critical entry points, or add it to a top-level instrumentation file. Fix the schema to match the actual env var names used.

### H2. 14 API routes have no try/catch error handling

**Files:** See list below
**What:** These routes have no top-level try/catch, meaning any unexpected error (DB connection failure, malformed input edge case) returns a raw Next.js 500 error page or an opaque error response:
- `src/app/api/admin/trails/[id]/status/route.ts`
- `src/app/api/affiliate/track/[slug]/route.ts`
- `src/app/api/auth/resend-verification/route.ts`
- `src/app/api/feed/click/route.ts`
- `src/app/api/forum/notifications/read/route.ts`
- `src/app/api/forum/notifications/route.ts`
- `src/app/api/forum/posts/[id]/route.ts`
- `src/app/api/forum/threads/[id]/route.ts`
- `src/app/api/marketplace/listings/[slug]/related/route.ts`
- `src/app/api/marketplace/listings/[slug]/route.ts`
- `src/app/api/marketplace/listings/[slug]/watch/route.ts`
- `src/app/api/cron/fantasy/open-rosters/route.ts`

**Why it matters:** Unhandled exceptions in API routes return unhelpful error responses to clients and may leak stack traces in development. The cron route `open-rosters` is particularly concerning because a failure there could silently prevent fantasy rosters from opening.
**Suggested fix:** Add a consistent try/catch wrapper pattern across all routes. Consider creating a shared `withErrorHandler` higher-order function.

### H3. 21 route groups missing error.tsx and loading.tsx boundaries

**What:** The following route groups have no `error.tsx` or `loading.tsx`:
`admin`, `bikes`, `creators`, `dashboard`, `donate`, `fantasy`, `feed`, `media`, `merch`, `messages`, `news`, `notifications`, `onboarding`, `parks`, `profile`, `register`, `search`, `shops`, `signin`, `verify-email`, `wallet`

**Why it matters:** Without `error.tsx`, an unhandled error in any of these sections bubbles up to the nearest parent error boundary (the root, if none exists), potentially crashing the entire page layout. Without `loading.tsx`, navigating to these pages shows no feedback while server components fetch data, creating a perceived "frozen" UI.
**Suggested fix:** Add at minimum a generic `error.tsx` and `loading.tsx` to each top-level route group. These can share a common component.

### H4. Rate limiter creates a new Redis client and Ratelimit instance on every call

**File:** `src/lib/rate-limit.ts`, lines 29-43
**What:** Every invocation of `rateLimit()` dynamically imports `@upstash/ratelimit` and `@upstash/redis`, creates a new `Redis` client instance, and creates a new `Ratelimit` instance. None of these are cached or reused.
**Why it matters:** Creating a new HTTP client and rate limiter on each call adds unnecessary overhead and may cause connection exhaustion under high load. The dynamic `import()` also adds latency on every call.
**Suggested fix:** Move the Redis client and Ratelimit instantiation to module scope (lazily initialized once) and reuse across calls. Use a module-level variable pattern similar to the Prisma singleton in `db/client.ts`.

### H5. Search queries use ILIKE (`contains: mode 'insensitive'`) without text search indexes

**File:** `src/lib/search/queries.ts`
**What:** All search functions (`searchThreads`, `searchListings`, `searchMembers`, `getGlobalSearchCounts`) use Prisma's `contains` with `mode: 'insensitive'`, which translates to `ILIKE '%query%'` in PostgreSQL. The schema has no `@@index` entries for the searched columns (`title`, `body`, `description`, `username`, `name`).
**Why it matters:** `ILIKE '%text%'` performs a full sequential table scan because it cannot use B-tree indexes (the leading `%` prevents index usage). As the data grows, search becomes progressively slower. The `getGlobalSearchCounts` function is especially expensive because it runs three separate count queries with full table scans in parallel on every search.
**Suggested fix:** Add PostgreSQL `pg_trgm` GIN indexes for the searched columns, or better yet, implement full-text search using `tsvector`/`tsquery` with a GIN index. For the count queries, consider caching or estimating counts.

### H6. Geospatial distance query in events uses expensive Haversine formula without PostGIS

**File:** `src/modules/events/lib/queries.ts`, lines 421-441
**What:** The `getEventsNearLocation` function calculates distances using a raw SQL Haversine formula (`acos(cos(...))`) on every row. While there is a `@@index([latitude, longitude])` on the events table, a B-tree index on two separate float columns cannot accelerate trigonometric distance calculations.
**Why it matters:** The query performs the expensive trigonometric calculation on every published event with coordinates before filtering. This becomes a full table scan in terms of computation cost.
**Suggested fix:** Use a bounding-box pre-filter (WHERE latitude BETWEEN ... AND longitude BETWEEN ...) to narrow the candidate set before applying the Haversine formula, or install PostGIS and use `ST_DWithin` for proper spatial indexing.

## Medium Priority Issues

### M1. 27 accidental duplicate files in the codebase

**What:** 27 files with a " 2" suffix exist in the codebase, concentrated in the trails and forum modules:
- `src/modules/trails/components/SystemClusterMap 2.tsx`
- `src/modules/trails/actions/toggleReviewHelpful 2.ts`
- `src/modules/trails/actions/getSystemsInBounds 2.ts`
- `src/modules/trails/actions/getSystemTrails 2.ts`
- `src/modules/forum/components/PostDetail 2.tsx`
- `src/modules/forum/components/NotificationBell 2.tsx`
- `src/modules/forum/lib/link-preview 2.ts`
- `src/modules/forum/lib/notifications 2.ts`
- `src/modules/profile/components/AvatarUpload 2.tsx`
- Plus 18 more similar files

**Why it matters:** Dead files increase cognitive load, can confuse IDE autocompletion, inflate bundle analysis, and may cause confusion about which version is canonical. The `" 2"` suffix pattern suggests accidental Finder-based file duplication.
**Suggested fix:** Delete all `" 2"` files after confirming they are identical or older than their counterparts.

### M2. `expireListings` server action has no auth guard but modifies data

**File:** `src/modules/marketplace/actions/maintenance.ts`
**What:** The `expireListings()` function is marked with `'use server'` and performs an `updateMany` to expire listings. It has no auth check.
**Why it matters:** Server Actions are publicly callable endpoints. A malicious actor could invoke `expireListings()` directly to expire all active listings whose `expiresAt` is in the past. While this matches what the cron job does, it should not be callable by unauthenticated users at will -- it could be used for timing attacks or to force-expire listings prematurely if the `expiresAt` check has edge cases.
**Suggested fix:** Either add `requireAdmin()` to protect it, or refactor it to a plain function (not a server action) that is only called by the cron route.

### M3. `Listing.seller` relation has no `onDelete` cascade rule

**File:** `prisma/schema.prisma`, line 1255
**What:** The `Listing` model's relation to `User` via `sellerId` has no `onDelete` specification: `seller User @relation(fields: [sellerId], references: [id])`. Same for `Transaction.buyer` and `Transaction.seller` (lines 1363-1365).
**Why it matters:** If a user account is deleted, their listings and transactions become orphaned records with dangling foreign keys. Prisma's default behavior is to reject the delete (restrict), which could prevent account deletion entirely or require manual cleanup.
**Suggested fix:** Add `onDelete: SetNull` (if you want to preserve the listing data) or `onDelete: Cascade` (if listings should be removed with the user). For transactions, `SetNull` is likely more appropriate to preserve the financial record.

### M4. 20 relations missing explicit `onDelete` rules

**What:** Across the schema, 20 foreign key relations have no `onDelete` specification, defaulting to Prisma's `Restrict` behavior. Notable ones include:
- `Post.author` (line 262) -- deleting a user fails if they have posts
- `Comment.author` (line 297) -- deleting a user fails if they have comments
- `Report.post` and `Report.comment` (lines 403-404) -- deleting content fails if reported
- `Trail.system` (line 685) -- deleting a trail system fails if it has trails
- `LearnQuizAttempt.quiz` (line 546) -- deleting a quiz fails if it has attempts

**Why it matters:** Without explicit cascade rules, user deletion and content moderation operations will fail with foreign key constraint errors. This creates a poor admin experience and can leave the system in inconsistent states.
**Suggested fix:** Review each relation and add appropriate `onDelete` rules. Common patterns: `Cascade` for content owned by a parent, `SetNull` for optional references, `Restrict` only where the referential integrity is intentionally enforced.

### M5. User search exposes banned users

**File:** `src/lib/search/queries.ts`, lines 71-89
**What:** The `searchMembers()` function queries all users matching the search term with no filter for `bannedAt`. Banned users appear in search results alongside active members.
**Why it matters:** Banned users should not be discoverable through search, as this undermines the ban's intent and may expose profiles that were banned for harassment or policy violations.
**Suggested fix:** Add `bannedAt: null` to the `where` clause.

### M6. `pool.max: 2` is very low for a production application

**File:** `src/lib/db/client.ts`, line 23
**What:** The PostgreSQL connection pool is configured with `max: 2` connections.
**Why it matters:** With only 2 connections available, any concurrent serverless function invocations beyond 2 will queue waiting for a connection. The fantasy lock-rosters cron job (which acquires connections in a loop) could monopolize both connections. Vercel serverless functions can have many concurrent instances.
**Suggested fix:** Increase to at least 5-10 for the session pooler. Note: the Supabase session pooler supports higher connection limits than the direct connection. Monitor connection usage and adjust based on actual concurrency.

### M7. Multiple large components exceed 300 lines with mixed responsibilities

**Files:**
- `src/modules/marketplace/actions/offers.ts` (644 lines) -- offer creation, acceptance, rejection, countering, withdrawal, and listing all in one file
- `src/modules/forum/lib/queries.ts` (579 lines) -- all forum read queries in one file
- `src/modules/events/lib/queries.ts` (484 lines) -- all event queries
- `src/modules/marketplace/components/sell/CreateListingForm.tsx` (476 lines) -- form with 25+ useState hooks
- `src/modules/trails/lib/queries.ts` (471 lines) -- all trail queries
- `src/app/trails/explore/[systemSlug]/page.tsx` (443 lines) -- page component with significant data transformation logic

**Why it matters:** Large files are harder to review, test, and maintain. The `CreateListingForm.tsx` with 25+ individual `useState` hooks is a strong candidate for `useReducer` or a form library.
**Suggested fix:** Split query files by domain entity. Extract the form state management in `CreateListingForm` to a custom hook or use a form library like react-hook-form.

### M8. `any` type usage in auth config

**File:** `src/lib/auth/config.ts`, line 41
**What:** The session callback uses `any` typing: `async session({ session, token }: any)`. This bypasses TypeScript's type safety for one of the most security-critical parts of the application.
**Why it matters:** Type errors in the session callback silently pass -- for example, accessing `session.user.role` when the shape has changed would not be caught at compile time.
**Suggested fix:** Import and use the proper NextAuth types: `Session` and `JWT` from `next-auth` and `next-auth/jwt`.

## Low Priority / Nice-to-Have

### L1. Raw `<img>` tags instead of `next/image` in 4 components

**Files:**
- `src/ui/components/fantasy/LeaderboardTable.tsx:33`
- `src/modules/learn/components/LessonViewer.tsx:115`
- `src/modules/bikes/components/BuildLogTimeline.tsx:29`
- `src/modules/bikes/components/BrandLinks.tsx:77`

**What:** These components use raw `<img>` HTML tags instead of Next.js `<Image>`.
**Why it matters:** Missing automatic image optimization (lazy loading, format conversion, responsive sizing). Minor performance impact per image, but adds up.
**Suggested fix:** Replace with `next/image` `<Image>` component.

### L2. 15+ instances of `unoptimized` prop on `<Image>` components

**What:** Many `<Image>` components use the `unoptimized` prop, which disables Next.js image optimization entirely.
**Why it matters:** These images are served at their original size and format, missing out on automatic WebP/AVIF conversion and responsive srcset generation. Particularly impactful for user-uploaded images and RSS feed article images.
**Suggested fix:** Configure `remotePatterns` in `next.config.ts` for any external image domains and remove `unoptimized`. For images on unknown domains (RSS feeds), `unoptimized` is acceptable but consider proxying through the Next.js image optimizer.

### L3. Console.log statements in production worker code

**Files:** `src/modules/fantasy/worker/mulliganAutoPick.ts` (6 instances), `src/modules/fantasy/worker/resultsScore.ts`, `src/modules/fantasy/worker/pricesReveal.ts`
**What:** Worker files use `console.log` for informational output rather than a structured logging approach.
**Why it matters:** In Vercel's production environment, `console.log` output goes to runtime logs but lacks structured metadata (severity level, timestamps, trace IDs). This makes it harder to filter and monitor worker activity.
**Suggested fix:** Replace with a structured logger or at least use `console.info` with consistent prefixes for easier filtering.

### L4. `generateSlug` / `slugify` duplicated across modules

**Files:**
- `src/modules/marketplace/actions/listing-mutations.ts:23` -- `slugify()`
- `src/modules/forum/actions/createThread.ts:29` -- `generateSlug()`
- `src/lib/slugify.ts` -- shared utility exists but is not always used

**What:** Two modules implement their own slug generation with slightly different logic, despite a shared `slugify.ts` utility existing in `src/lib/`.
**Why it matters:** Inconsistent slug formats across the application. One uses `Math.random().toString(36).substring(2, 8)`, the other `.slice(2, 8)` -- functionally similar but unnecessarily divergent.
**Suggested fix:** Use the shared `src/lib/slugify.ts` everywhere.

### L5. Feature flags defined in env.ts but not used through a feature flag system

**File:** `src/lib/env.ts`, lines 42-44
**What:** Feature flags (`FEATURE_COACHING`, `FEATURE_MARKETPLACE`, `FEATURE_MERCH`) are defined in the env schema but since `env.ts` is never imported, they are effectively dead code.
**Why it matters:** There is no actual feature gating happening -- all modules are always active regardless of these env vars.
**Suggested fix:** Either wire up the feature flags through a flags module (the file `src/lib/flags.ts` already exists -- check if it's connected) or remove them from the schema.

### L6. `iCal` server action lacks auth (acceptable but notable)

**File:** `src/modules/events/actions/ical.ts`
**What:** The `generateEventIcal` server action has no auth check. It generates an iCal file for any event by slug.
**Why it matters:** This is likely intentional (calendar files should be accessible to anyone viewing the event page), but worth noting that any server action without auth is a conscious design choice.

## Positive Observations

The audit also identified several well-implemented patterns worth preserving:

1. **All 25 `pool.query` calls use parameterized queries** (`$1`, `$2`, etc.) -- no SQL injection vectors found.
2. **All 10 cron routes verify `CRON_SECRET`** before executing.
3. **DOMPurify sanitization** is correctly applied before rendering user-supplied HTML in the news feed (only 1 instance in the entire codebase, properly sanitized with an allowlist of tags and attributes).
4. **Zod validation** is consistently used in server actions for input validation.
5. **The `requireAuth` guard** correctly checks for both authentication and ban status with proper redirects.
6. **The tipping system** (`/api/forum/posts/[id]/tip`) uses `FOR UPDATE` row-level locks and transactional credit deduction -- a solid pattern for financial operations.
7. **The module directory structure** is clean and well-organized with clear separation of concerns (actions, components, lib, queries).
8. **Webhook signature verification** is properly implemented for both Stripe endpoints.
9. **Rate limiting** is applied to user-facing mutation endpoints (chat, thread creation, tipping).
10. **The middleware auth guard** correctly checks for session cookies without making a DB call, keeping middleware fast.
